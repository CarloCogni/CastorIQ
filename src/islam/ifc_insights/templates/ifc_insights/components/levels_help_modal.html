How Levels Work

What is a Level?

A Level represents a floor or storey in your building model at a specific Z-elevation (mm above datum). Castor reads them from IfcBuildingStorey objects in your IFC file, lets you edit them, and can write them back to the file so your model stays in sync.

Lifecycle

From IFC Imported directly from IfcBuildingStorey — global ID is linked so Apply writes back to the same storey.
Suggested Derived from entity spatial containers when the IFC has no storey objects. Z values are synthetic (3 m increments).
Manual Created directly by you. Apply creates a new IfcBuildingStorey in the file.

How to populate levels

  1. Suggest from IFC — reads your uploaded IFC file and proposes storeys. Click Import next to any suggestion to add it.
  2. Add manually — type a name and Z elevation then click Add.
  3. Edit inline — click the pencil icon on any row to rename or adjust the elevation.
  4. Apply to IFC — writes all level records back into the IFC file, creating or updating IfcBuildingStorey objects. A .ifc.bak backup is made automatically before any write.

Who can do what

  • Any project member can view and add levels.
  • Apply to IFC modifies the IFC file on disk — treat it like a write-back operation.

Caveats

Apply overwrites the IFC file in place. A .ifc.bak is created first but is not versioned. Suggested Z-elevations for entity-derived storeys are synthetic (3 m per floor) — verify before applying. Geometric placement (slab shape, thickness) is out of scope — only the storey name and Z-elevation are written.
