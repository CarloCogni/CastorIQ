What can Castor modify?

Castor uses a Risk-Stratified Autonomous Action (RSAA) model. Instead of letting the LLM write directly to your IFC file, every request goes through a structured safety pipeline with human approval before anything is written.

Geometry is always out of scope. Castor modifies properties, attributes, classifications, and materials only. It does not move, resize, reshape, or create physical geometry. The only spatial operation available is level reassignment (Tier 3).
How every request is processed
1
Intent Classification
The LLM reads your message and classifies the intent: which tier, which operation, which entities, which properties, and with what values. It produces structured JSON — the IFC file is not touched.
2
Validation & Automatic Escalation
Castor validates the intent against your actual IFC data. If validation fails (e.g. property not found, pset missing), it auto-escalates to the next tier — always trying the safest tier first.
3
Guardian Check (RAV)
Before presenting a proposal, Castor cross-checks the intended change against your uploaded technical documents. This is advisory — it flags contradictions but never blocks the proposal.
4
Your Approval
You see a diff preview (Tier 1 & 2) or generated code (Tier 3) before anything is written. Nothing runs without your explicit approval.
5
Git Snapshot → Execute → Sync
On approval, a Git snapshot of the IFC file is taken, the change executes, then the database is synced. Every approved change is reversible from the History tab.
The three tiers
Tier What the LLM does What executes When used
1 GREEN Extracts structured intent (JSON) only Pre-coded, fully validated handlers — LLM never writes to the file Single property or attribute changes
2 ORANGE Generates an ordered operation plan (JSON steps) Each step validated independently before execution Multi-step ops, Pset management, materials, classifications
3 RED Generates IfcOpenShell Python code 7-layer sandbox + mandatory human code review Entity creation / deletion, spatial ops, relationships
Common point of confusion
  • Remove a property (e.g. "Remove FireRating from all walls") → Tier 1 REMOVE_PROPERTY. The property set survives; only that one key is deleted.
  • Remove an entire property set (e.g. "Remove Pset_Maintenance from all doors") → Tier 2 REMOVE_PSET. The whole Pset and all its properties are deleted.

Standard property sets (Pset_WallCommon, Pset_DoorCommon, etc.) are auto-created if they don't exist yet.

Remove a property REMOVE_PROPERTY

Removes a single property from within a property set — the Pset itself is preserved. To remove an entire property set, use Tier 2 → REMOVE_PSET.

Rename or change attributes SET_ATTRIBUTE

Name, Description, ObjectType, and Tag are direct IFC attributes — not properties inside a Pset.

Multiple changes at once CHAIN

Use "and" to combine simple changes. Each becomes a separate proposal you can approve together.

Tips for best results
  • Be specific about entity types — say "wall" not "element"
  • Use exact property names when you know them (e.g. "FireRating" not "fire rating")
  • Specify the floor/level to narrow scope: "...on Level 1"
  • Use entity names for single-entity changes: "wall W-01"
  • One change at a time works best — or use "and" for simple combos

Remove an entire property set REMOVE_PSET

Removes the entire property set and all properties it contains. To remove only a single property within a Pset, use Tier 1 → REMOVE_PROPERTY.

Castor decomposes complex requests into ordered steps. You review the full plan before approving.

Tips for Tier 2
  • Describe the goal, not individual steps — Castor will decompose for you
  • Mention the classification system name for classification ops
  • Material names should match your project conventions (e.g. "Concrete C30/37")
  • You can mix property changes with structural ops in one request
  • Review each step in the plan — you approve the whole plan at once

Safety notes
  • Generated code runs on a copy of your IFC file — the original is safe until you confirm
  • A Git snapshot is taken before execution — you can always restore from History
  • Read the generated code before typing CONFIRM
  • If the code looks wrong, reject and rephrase your request
  • Complex operations may have lower confidence — this is expected
