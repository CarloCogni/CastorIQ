How Conflict Detection Works
{# Body #}
{# Intro #}

Castor runs two complementary checks to ensure your IFC model is internally consistent and aligned with your project documents.

{# ── Two check types side by side ── #}
{# Data Quality #}
Data Quality

Structural problems inside your IFC file. Three checks run on every upload: duplicate GlobalIDs (element skipped), missing property sets, and missing spatial placement (element imported with warning).

Runs automatically on every IFC upload Fix in your BIM tool (Revit, Bonsai, ArchiCAD…) and re-upload Affected elements are skipped during import
{# Semantic Conflicts #}
Semantic Conflicts

Contradictions between your IFC model and project documents — e.g. a fire rating mismatch or wrong material spec.

Runs on demand — you trigger the scan AI-powered: compares entity properties against doc requirements One-click fix through the Modify pipeline
{# ── How a semantic scan works ── #}

What happens during a scan?

1
Select entities — Castor picks IFC elements to check (physical elements by default, or all if you uncheck "Skip non-physical elements").
2
Find relevant documents — Vector search matches each entity with the most relevant sections from your uploaded documents.
3
AI comparison — The LLM checks whether properties match requirements and flags contradictions with a severity level (critical, high, medium, low).
4
Results appear here — Each conflict shows the IFC value vs. the document value, with a link to fix it or dismiss it.
{# ── Workflow strip ── #}

Typical workflow

Upload IFC + Docs Data Quality auto-checked Run Semantic Scan Review conflicts Fix in Modify
{# ── Tips ── #}

Tips

Re-scan after uploading new documents or modifying the IFC model. Dismissed conflicts won't reappear unless a future scan re-detects them. Scan duration depends on project size — typically 30 seconds to a few minutes. "Fix in Modify" opens the Modify tab pre-filled with a suggested correction.
