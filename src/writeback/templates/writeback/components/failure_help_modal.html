Why did this modification fail?

Castor runs all language-model inference locally, on your own machine, through Ollama. That choice protects your data and works offline, but it comes with a tradeoff: the locally-runnable models are smaller than frontier APIs, and on complex prompts they sometimes produce malformed or incomplete output. Castor catches those failures cleanly instead of acting on garbage.

What the failure types mean
Request Ambiguous

The model produced a structured intent but couldn't extract a target value from your message — for example, it set new_value to null. This often happens when an element name contains unusual characters (colons, embedded numbers, mixed punctuation) and the model loses focus partway through. Fix: retype the value clearly, e.g. set FireRating to EI240 on wall <name>, or switch to a more capable model.

Request Too Vague

The request didn't carry enough information to identify a target. Fix: name the entity type, the property, and the new value explicitly.

Filter No Match / Filter Empty

The targeting filter the model produced didn't match any entity in the IFC file, or matched everything (refused as unsafe). Fix: include a more specific identifier — element name, level, or a property the target entity has.

Property Not Found / Pset Not Found

The property or pset the model named doesn't exist on the target entities. Fix: use exact IFC names (e.g. Pset_WallCommon.FireRating, not fire-rating).

Model size matters

Reliability of these structured-output tasks scales with model size. In our testing:

  • 7–8B models (e.g. llama3.1:8b) drift frequently on complex prompts: they invent field names, drop values, or wrap output in unexpected nesting. Castor's repair layer recovers many of these, but not all.
  • 14B+ models (e.g. qwen3:14b) handle the same prompts reliably. If you keep hitting the same failure, try switching from the model selector at the top of the page.
Why complex element names trip the model

Revit-style names like Basic Wall:Wall-Ext_102Bwk-75Ins-100LBlk-12P:285330 contain colons, hyphens, embedded numerics, and a numeric suffix that resembles a code identifier. Small models often misclassify these as code rather than a name string and produce inconsistent output around them. If you can identify the wall by a simpler attribute (level, type, role), prefer that.

If the same prompt keeps failing
  1. Switch the model from the selector (try a 14B+ model).
  2. Rephrase: put the verb and value first, then the target (e.g. "Set FireRating to EI240 on the wall named …").
  3. Identify the target by a simpler attribute when possible (level, name pattern, type) instead of a full Revit name.
