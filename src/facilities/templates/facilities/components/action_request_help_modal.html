How Action Requests Work

What is an action request?

An action request is a lightweight issue report — the entry point for anyone with project access (typically a tenant or occupant) to flag something. It carries the bare minimum to triage: title, description, optional asset or space anchor, and a severity. The FM team decides whether to escalate it into a work order, dismiss it, or simply mark it triaged while investigating.

Lifecycle

Open Freshly submitted — waiting on the FM team for a first look.
Triaged Acknowledged by the FM; being investigated or awaiting input. The triage note captures context.
Escalated Promoted into a work order — the new WO links back to this request via source_action_request so the provenance survives.
Dismissed Closed without action — duplicate, not applicable, or out of scope. A reason goes into the triage note.

Escalating to a work order

The Escalate to Work Order button on the request detail creates a new WO with the request's title, description, and any asset / space anchors pre-filled; the request status moves to escalated in the same transaction. Escalation is still gated by Editor+ access — tenants cannot escalate their own requests.

Who can do what

Anyone with project access may submit a request via the Report an issue drawer. Editor / Owner may triage, dismiss, escalate, or edit requests. Submitters see their own requests; viewers see the request list read-only.

Caveats

An action request is not a work order — there is no assignee, no schedule, no SLA clock. Escalate to a WO when any of those are needed. Escalation is one-way: a request can't be un-escalated, but the generated WO can still be cancelled. The richer occupant-portal UX (mobile-first, LLM-aided intake) lands in M4; this form is the interim triage surface.
